# Linkit 项目宪法（Constitution）

> 文件路径：`docs/spec/constitution.md`  
> 版本：1.0.0（正式）  
> 日期：2026-07-16

---

## 项目说明

Linkit 是一款面向 Windows 与 macOS 的桌面端智能知识收藏应用，用于收集、整理、发现和再利用网址、网页资源与灵感素材。

本宪法规定 Linkit MVP 的技术栈、工程实践、安全要求、测试基线和规格文档职责。所有实现、审查、测试与交付活动均须遵守本文件；具体功能范围与验收标准由 `docs/spec/requirements.md` 定义。

---

## 技术栈

### 后端与桌面运行时

- **主语言**：Go 1.26.x。
- **桌面框架**：Wails v2 稳定版本线。
- **目标平台**：Windows、macOS，两个平台的核心功能必须对等。
- Wails 的具体补丁版本必须在 `go.mod` 中锁定，不得使用 alpha、beta 或 rc 版本。
- 正式交付以 Wails 桌面构建产物为准；Vite 独立预览仅用于前端开发和原型验证。

### 前端与云服务

- **运行时**：Node.js 24.x。
- **包管理器**：pnpm 10.x。
- **前端框架**：React 18.3.1。
- **语言**：TypeScript 5.5.3。
- **构建工具**：Vite 5.4.2。
- **样式方案**：Tailwind CSS 3.4.1。
- **云服务**：Supabase，客户端 SDK 基线为 `@supabase/supabase-js` 2.57.4。
- 已锁定的基线版本如需升级，必须先说明升级原因、兼容性影响、迁移步骤与回滚方案。

### TypeScript 规范

- 必须启用严格模式（`strict: true`）。
- 禁止使用显式或隐式 `any`。
- `unknown` 仅用于外部输入、JSON、第三方 SDK 返回值等不可信边界，并且必须在使用前完成类型收窄或运行时校验。
- 允许使用 `as const`。
- 类型断言仅允许出现在缺少可靠类型信息的边界位置，必须保持最小作用域并具有可验证依据。
- 禁止双重断言，例如 `value as unknown as TargetType`。
- 禁止使用 `@ts-ignore`、无说明的 `@ts-expect-error`、`as any` 或无说明的 ESLint 抑制注释绕过检查。
- 每个前端任务完成后必须执行 `pnpm typecheck` 与 `pnpm lint`。

## 包管理规范

1. 项目统一使用 pnpm，禁止混用 npm 或 yarn 安装依赖。
2. `pnpm-lock.yaml` 是唯一的 Node.js 依赖锁文件，必须提交到版本控制。
3. 不得保留或提交 `package-lock.json`、`yarn.lock` 等其他包管理器锁文件。
4. `package.json` 必须通过 `packageManager` 字段锁定 pnpm 主版本与经验证的工具版本。
5. CI 必须使用冻结锁文件安装依赖，不得在流水线中隐式更新锁文件。
6. 直接依赖使用已验证的稳定版本；禁止引入 alpha、beta、rc 或来源不明的软件包。
7. 新增依赖前必须确认现有代码或标准库无法合理满足需求，并记录用途、安全影响与维护成本。
8. Go 依赖必须由 `go.mod` 和 `go.sum` 管理并提交到版本控制。

## 工程化原则

1. **严禁重复**：重复逻辑必须抽象为职责明确、可复用的函数、组件、Hook、Service 或模块。
2. **先检查现有实现**：新增功能前必须搜索并评估已有代码，避免平行实现。
3. **模块化设计**：界面、业务规则、数据访问、外部服务与平台能力必须保持清晰边界。
4. **统一接口**：相同类别的函数、服务和数据操作必须采用一致的参数与返回值结构。
5. **集中管理配置**：常量与非敏感配置统一放入对应模块的 `config/` 目录；环境相关配置不得散落在业务代码中。
6. **优先封装**：外部 API、Supabase、持久化、AI 调用和 Wails 平台能力必须通过独立适配层访问。
7. **禁止硬编码环境信息**：密钥、服务地址、用户数据与环境差异必须由环境变量或本机配置提供。
8. **界面与错误语言**：代码注释和测试描述使用中文；默认 UI 文案、错误提示和日志消息使用英文，并通过 i18n 提供中文翻译。

## 编码安全

### Supabase 行级安全（RLS）

- 所有包含用户数据的 Supabase 表必须启用 RLS。
- 用户数据策略必须以 `auth.uid()` 为隔离依据，确保用户只能执行本人数据的 `SELECT`、`INSERT`、`UPDATE` 与 `DELETE`。
- RLS 策略必须通过真实的授权与越权用例验证；仅在客户端过滤数据不构成安全边界。
- Service Role Key 不得进入桌面客户端、前端构建产物或用户可读取的本机配置。

### Rate Limiting

- 当前桌面客户端直接调用的第三方 API 遵守服务提供方的限流与重试规则。
- 未来新增的自建 HTTP API 或 Supabase Edge Function 默认执行每 IP 每小时 100 次请求的限流策略。
- 业务需要不同阈值时，必须通过集中配置调整，并在接口规格中说明理由。
- 超限响应必须提供清晰的英文错误信息，不得泄露内部实现或敏感信息。

### API 密钥与敏感数据

- 构建期密钥存储在 `.env` 或 `.env.local` 中，并确保相关文件已加入 `.gitignore`。
- 桌面端用户配置的 AI API Key 只能存储在本机安全设置中，不得上传至 Supabase 或写入日志。
- 代码必须通过受控配置模块读取环境变量，禁止在业务模块中直接散落访问逻辑。
- 禁止向版本控制提交 API Key、Supabase 密钥、测试账号密码、令牌或真实用户数据。
- 日志、错误报告、截图和测试证据必须对密钥、令牌与个人信息进行脱敏。

### 破坏性操作

- 存储模式切换、导入覆盖、删除和去重合并等破坏性操作必须获得用户明确确认。
- 默认行为不得静默覆盖、删除或迁移用户数据。
- 数据变更必须提供明确的源、目标与影响说明；失败时不得造成部分状态被误报为成功。

## 质量基线

### 静态检查

- 前端代码必须通过 ESLint 与 TypeScript 类型检查。
- Go 代码必须通过 `gofmt`、`go vet` 和相关测试。
- 不得通过降低规则等级、全局忽略或删除测试来规避质量门禁。

### Git Hooks

- 使用 Husky 与 lint-staged 建立提交前检查。
- 提交前至少检查受影响文件的格式、Lint 与类型错误。
- Git Hooks 是本地反馈机制，不能替代 CI 的完整检查。

### CI/CD

- 使用 GitHub Actions 执行自动化质量检查和安全扫描。
- CI 必须覆盖依赖安装、Lint、类型检查、Go 测试、React 测试和生产构建。
- 桌面构建必须至少覆盖 Windows 与 macOS。
- 依赖与安全扫描发现高严重度问题时不得发布。
- CI、测试框架或发布流程的重大调整必须先更新技术设计或任务规格。

### 完成标准

- 未真实执行的测试不得标记为通过。
- 无法执行的测试必须标记为 `BLOCKED`，并记录原因、影响和解除条件。
- 每个实现任务必须绑定 TASK、REQ、AC、测试类型与可复核的验收证据。

## 测试策略

### TDD 工作流

- 所有生产代码遵循 Red、Green、Refactor 循环。
- Red：先编写失败测试，并真实执行以确认测试因缺少目标行为而失败。
- Green：编写使测试通过的最小实现。
- Refactor：在测试保持通过的前提下消除重复、改善命名与模块边界。
- 禁止先完成生产实现后补写无法证明驱动过程的测试。

### 测试工具

- **Go 单元与集成测试**：`go test`，涉及核心逻辑时输出覆盖率。
- **React 单元测试**：Vitest。
- **React 组件测试**：React Testing Library + Vitest，覆盖渲染、交互与可访问性。
- **桌面端 E2E**：Playwright，覆盖 Wails 开发模式或打包应用的关键业务路径。
- **视觉回归**：Playwright Screenshot，UI 变更必须保留 Baseline、实际截图与 Diff。
- **API 测试**：简单接口使用 `curl`，复杂接口使用 Go Integration Test。
- **性能测试**：Go 核心算法与数据处理使用 `go test -bench`。

### 测试数据

- 测试账号和测试数据统一记录在 `docs/spec/info.md`。
- 禁止自行捏造测试账号、密钥、服务返回值或测试通过结果。
- AI 功能必须分别验证有效配置路径和无 Key、网络失败或服务失败时的降级路径。

## 文档结构

```text
docs/spec/
├── constitution.md          # 本文件，定义不可绕过的项目工程基线
├── requirements.md          # 使用 EARS 描述的需求与 REQ/AC 验收标准
├── design.md                # 架构、模块边界和关键技术决策
├── data.md                  # 数据实体、存储结构、迁移与 RLS 设计
├── api.md                   # 内部接口、Wails 绑定和外部 API 契约
├── tasks.md                 # 绑定 REQ/AC 的可执行任务清单
├── test_strategy.md         # 测试分层、工具、环境与质量门禁
├── info.md                  # 经授权的测试账号与测试数据
├── traceability.md          # REQ、AC、TASK、测试与证据追溯矩阵
├── analysis-report-*.md     # 规格一致性审查报告
├── ac/                      # 各任务的 AC 验收矩阵
│   └── TASK-XXX-AC.md
└── reports/                 # 实际执行产生的测试与验收报告
    └── TASK-XXX-test-report.md
```

- `constitution.md` 定稿后不得静默修改；任何变更必须说明原因、影响范围和版本变化。
- 需求、设计、数据或 API 发生变化时，必须按 SDD 规格回退协议更新下游任务与追溯关系。
- 规格文档不得以代码现状替代；代码、测试和文档发生冲突时必须先明确应回退的 SDD 阶段。

## 宪法治理

1. 本文件适用于 Linkit 的规格、设计、实现、测试、审查与发布活动。
2. 当前版本已经用户明确二次确认并正式生效；后续不得静默修改。
3. 定稿后的修改必须递增版本号，并在文末记录日期、修改原因和影响范围。
4. 与本宪法冲突的任务或实现必须暂停，不得通过临时例外静默绕过。
