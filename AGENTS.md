## 沟通规范

1. 所有内容必须使用中文交流（包括代码注释），但是文案与错误提示要使用英文。
2. 遇到不清楚、歧义、缺失的内容应立即向用户提问。
3. 表达清晰、简洁、技术准确。
4. 在代码中应添加必要的注释解释关键逻辑。
5. 当前任务完成后，提出下一步执行的建议。

## 开发工程化规范

在编写代码时，请务必遵循以下原则：

1. 严禁重复: 任何重复的代码逻辑都必须被抽象成可复用的单元（函数、类等）。
2. 检查现有代码：先分析项目中是否已有类似实现
3. 模块化设计：将功能拆分为独立、可复用的模块
4. 统一接口：使用一致的参数格式和返回值结构
5. 配置集中管理：所有常量和配置项统一管理在 `config/` 目录
6. 优先封装: 将相关功能和数据封装在独立的、职责明确的模块中。
7. 环境变量禁止硬编码

## 开发方式

严格遵守测试驱动开发（TDD）流程。请严格遵循红-绿-重构循环。

* 先分析需求并拆分测试点
* 先编写失败的单元测试（Red）
* 编写最小实现使测试通过（Green）
* 在测试保护下重构代码（Refactor）

## 执行任务

1.执行复杂问题时使用 Sequential Thinking mcp 工具，拆分任务。

2.使用context7 mcp 工具，查询对应的文档。

3.涉及ui设计的任务时使用Playwright mcp （Playwright Screenshot）工具进行视觉回归测试。

4.当运行 `find` 或类似命令时，要注意 `node_modules`、`vendor `并忽略它。

5.使用context7 mcp 查询框架、包等的最新文档。

6.当使用supabase时，使用supabase mcp进行连接。

## QA 测试执行规范

| 测试层级           | 推荐工具                                     | 适用场景                                       | 输出要求                      |
| ------------------ | -------------------------------------------- | ---------------------------------------------- | ----------------------------- |
| **Go 单元测试**    | `go test`                                    | Go 后端业务逻辑、Service、Repository           | 输出测试结果及覆盖率          |
| **React 单元测试** | Vitest                                       | Hooks、Utils、状态管理等                       | 输出覆盖率报告                |
| **React 组件测试** | React Testing Library + Vitest               | 组件渲染、交互、可访问性                       | 覆盖主要交互路径              |
| **桌面端 E2E**     | Playwright                                   | Wails 桌面应用完整业务流程（开发模式或打包版） | **必须截图留证**              |
| **视觉回归测试**   | Playwright Screenshot（Snapshot）            | UI 布局、主题、组件样式、关键页面              | **必须生成 Baseline 与 Diff** |
| **API 接口测试**   | `curl`（简单） / Go Integration Test（复杂） | Go Backend API、HTTP 接口                      | 展示请求与实际返回值          |
| **性能基准测试**   | `go test -bench`                             | Go 核心算法、数据处理                          | 输出 Benchmark 数据           |
| **代码覆盖率**     | Go Cover + Vitest Coverage（V8）             | Go 与 React 覆盖率统计                         | 自动生成 Coverage Report      |

### 4.1 禁止伪造测试结果

> ⛔ 严禁在未执行真实调用、未获取实际返回值的情况下输出 `PASS`。

* 无法执行的用例须标记 `BLOCKED` 并说明原因。
* 不得跳过、伪造、捏造任何测试结论。
